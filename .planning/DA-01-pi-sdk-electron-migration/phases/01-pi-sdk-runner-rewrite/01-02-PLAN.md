# PLAN 01-02 — Rewrite planningRunner.ts with Pi SDK + Pi-Native Skills

## Objective
Replace the `gnome-terminal` + `opencode-cli run` approach with an in-process Pi SDK `AgentSession` that invokes pi-native skills (`/skill:start-work-begin` + `/skill:start-work-plan`) for structured planning. Planning sessions run entirely within the Node.js process, stream events to the UI, and produce the same `.planning/` directory artifacts.

## Key Design Decisions

### Skill invocation
The planning runner no longer manually constructs a planning prompt that describes the `start-work` + `create-plans` workflow. Instead, it sends skill commands directly through the Pi session:

1. **Scaffold phase:** Send `/skill:start-work-begin` with the task context (identifier, title, description, work type, relevant files). This produces BRIEF.md, ROADMAP.md, and phase stubs.
2. **Deepen phase:** Send `/skill:start-work-plan` to expand each phase from the roadmap into detailed PLAN.md files.

The Pi SDK expands skill commands automatically when sent via `session.prompt()`.

### Tools for planning
Planners should NOT modify source code. They write to `.planning/` only:
- Use `createReadOnlyTools(repoPath)` as the base (read, grep, find, ls)
- Add a custom `bash` tool so planner can explore the repo (git log, file listings, etc.)
- Skills themselves handle writing to `.planning/` via their own instructions — the model uses the available tools as guided by the skill

### System prompt
Use the persona's `systemPrompt` field as the base. The skill content is appended by Pi's skill discovery system — no manual prompt construction needed.

### Session management
Use `SessionManager.inMemory()` — planning sessions are ephemeral within Dash AI's task lifecycle.

## Tasks

### 1. Rewrite `runPlanningSession()` in `planningRunner.ts`

```typescript
import {
  createAgentSession,
  SessionManager,
  createReadOnlyTools,
  DefaultResourceLoader,
  SettingsManager,
} from "@mariozechner/pi-coding-agent"
import { resolveModel, checkProviderAuth } from "./piSession"

export async function runPlanningSession(
  input: PlanningRunnerInput,
  onEvent: (type: string, payload: Record<string, unknown>) => Promise<void>
): Promise<PlanningResult> {
  try {
    await onEvent("PLANNING_EVENT", { status: "starting", message: "Starting planning session" })

    // Auth pre-flight
    const providerID = input.planningPersona.provider || "anthropic"
    const authCheck = await checkProviderAuth(providerID)
    if (!authCheck.ok) throw new Error(authCheck.errorMessage!)

    // Resolve model via Pi registry
    const model = await resolveModel(providerID, input.planningPersona.model)

    await onEvent("PLANNING_EVENT", {
      status: "launching",
      message: "Launching planning session",
      model: `${model.provider}/${model.id}`,
    })

    const loader = new DefaultResourceLoader({
      cwd: input.repoPath,
      // System prompt from persona — Pi appends skill content automatically
      systemPromptOverride: () => input.planningPersona.systemPrompt,
    })
    await loader.reload()

    const { session } = await createAgentSession({
      cwd: input.repoPath,
      model,
      thinkingLevel: "high",
      tools: createReadOnlyTools(input.repoPath),
      resourceLoader: loader,
      sessionManager: SessionManager.inMemory(),
      settingsManager: SettingsManager.inMemory({ compaction: { enabled: true } }),
    })

    // Subscribe to events and forward to Dash AI's event system
    session.subscribe((event) => {
      switch (event.type) {
        case "message_update":
          if (event.assistantMessageEvent.type === "text_delta") {
            onEvent("PLANNING_TEXT", { delta: event.assistantMessageEvent.delta })
          }
          break
        case "tool_execution_start":
          onEvent("TOOL_START", { toolName: event.toolName, args: event.args })
          break
        case "tool_execution_end":
          onEvent("TOOL_END", { toolName: event.toolName, isError: event.isError })
          break
        case "turn_start":
          onEvent("TURN_START", { turnIndex: event.turnIndex })
          break
        case "turn_end":
          onEvent("TURN_END", { turnIndex: event.turnIndex })
          break
        case "agent_end":
          onEvent("PLANNING_EVENT", { status: "completed" })
          break
      }
    })

    // Phase 1: Scaffold the work item using start-work-begin
    const workName = input.taskTitle
      .toLowerCase()
      .replace(/[^a-z0-9]+/g, "-")
      .replace(/^-|-$/g, "")
      .slice(0, 40)

    await session.prompt(
      [
        `/skill:start-work-begin`,
        `Identifier: ${input.taskIdentifier || "(auto-generate)"}`,
        `Work name: ${workName}`,
        `Work type: Feature`,
        `Description: ${input.taskDescription}`,
        input.targetFiles?.length ? `Relevant files: ${input.targetFiles.join(", ")}` : "",
      ].filter(Boolean).join("\n")
    )

    // Phase 2: Deepen the roadmap into executable plans using start-work-plan
    // The skill will walk through the ROADMAP.md phases and create PLAN.md files
    await session.prompt(
      `/skill:start-work-plan — expand all phases in the roadmap into detailed PLAN.md files`
    )

    // Find the plan docs pi wrote under .planning/
    const planDocPath = findLatestPlanDir(input.repoPath)

    session.dispose()

    return { success: true, planDocPath }
  } catch (error) {
    // ... error handling
  }
}
```

### 2. Handle plan iteration (feedback loop)
When a user clicks "Iterate Plan" in the dashboard, Dash AI sets the task back to `IN_PLANNING` with `planFeedback` text. The planning runner should include this feedback:

```typescript
// For iteration: the .planning/ scaffold already exists, just re-plan with feedback
if (input.planFeedback) {
  await session.prompt(
    [
      `/skill:start-work-plan`,
      `The user reviewed the plan and provided this feedback:`,
      ``,
      input.planFeedback,
      ``,
      `Refine the affected phase plans accordingly.`,
    ].join("\n")
  )
}
```

This is a major improvement over the current approach, which re-runs the entire `start-work` + `create-plans` pipeline. With `start-work-plan`, we can target specific phases for refinement.

### 3. Remove all `gnome-terminal` and `opencode-cli` code
- Delete the `spawn("gnome-terminal", ...)` block
- Delete the temp shell script creation
- Delete the process exit polling / heartbeat interval
- Delete `normalizeModel()` function
- Remove `buildPlanningPrompt()` — skill invocation replaces the manually-constructed prompt
- Keep `findLatestPlanDir()` helper (still needed to locate generated plan docs)

### 4. Remove `checkSkillsInstalled()`
- Pi's `DefaultResourceLoader` discovers skills from `~/.agents/skills/` natively
- No need for manual filesystem checks
- If skills are missing, Pi will emit an error event that we can surface to the UI
- The startup warning in `index.ts` will be updated in 01-04

### 5. Session timeout
- Add optional `setTimeout` wrapper that calls `session.abort()` after 20 minutes
- Simpler than current approach — no polling interval needed
