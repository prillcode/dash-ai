# PLAN 01-04 ‚Äî Update queueWorker and Clean Up

## Objective
Adapt the queue worker to manage Pi SDK sessions instead of spawned processes, and clean up all remaining OpenCode references.

## Tasks

### 1. Update `queueWorker.ts`

**Changes:**
- Remove `runSession` import from `sessionRunner` ‚Äî import `runCodingSession` directly from `codingRunner`
- The `runPlanningTaskSession()` and `runTaskSession()` functions stay structurally the same
- They already call `runPlanningSession()` and `runSession()` respectively ‚Äî just update the imports
- Remove any references to `gnome-terminal` process management
- The concurrent session limit (`MAX_CONCURRENT_SESSIONS`) now limits in-process Pi sessions ‚Äî this is a real resource constraint (memory, API rate limits), not just a process count

**Concurrency consideration:**
Pi SDK sessions are in-process. Multiple concurrent sessions mean multiple LLM streaming connections. The existing `maxConcurrent` env var (default 3) is still appropriate. Consider reducing default to 2 for memory safety.

### 2. Simplify or remove `sessionRunner.ts`

The `SessionRunner` class and `runSession()` export are thin wrappers:
```typescript
// current: sessionRunner.ts wraps codingRunner.ts
export async function runSession(task, persona) {
  return await runCodingSession(...)
}
```

**Option A (recommended):** Remove `sessionRunner.ts` entirely, update `queueWorker.ts` to import from `codingRunner.ts` directly.

**Option B:** Keep as a thin adapter if there's value in the abstraction. But the indirection is unnecessary now that both runners use the same Pi SDK.

### 6. Update `packages/server/src/index.ts`
- Remove `import { checkSkillsInstalled } from "./agent/planningRunner"`
- Replace with a startup check for pi-native skills:
  ```typescript
  import { existsSync } from "fs"
  import { join } from "path"
  import { homedir } from "os"
  import { getAuth, getModelRegistry } from "./agent/piSession"

  // Verify pi-native skills are installed
  const requiredSkills = ["start-work-begin", "start-work-plan", "start-work-run"]
  const missing = requiredSkills.filter(
    (s) => !existsSync(join(homedir(), ".agents", "skills", s))
  )
  if (missing.length > 0) {
    console.warn(`‚öÝÔ∏è  Missing pi-native skills: ${missing.join(", ")}`)
    console.warn("    Planning and coding tasks will fail until skills are installed.")
    console.warn("    Expected location: ~/.agents/skills/")
  }

  // Verify Pi SDK can initialize auth + model registry
  try {
    const registry = getModelRegistry()
    const models = await registry.getAvailable()
    console.log(`Pi SDK initialized: ${models.length} models available`)
  } catch (err) {
    console.warn(`‚öÝÔ∏è  Pi SDK initialization warning: ${err.message}`)
  }
  ```
- Note: skill check remains a filesystem check because the skills are loaded per-session by `DefaultResourceLoader`, not at server startup. The check is a user-facing warning only.
