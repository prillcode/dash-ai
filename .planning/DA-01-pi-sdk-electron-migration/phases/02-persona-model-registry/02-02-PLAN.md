# PLAN 02-02 — Validate Persona Model/Provider Against Pi Registry

## Objective
When creating or updating a persona, validate the `model` and `provider` fields against Pi's `ModelRegistry` so that invalid model configurations are caught early — before a task fails at runtime.

## Tasks

### 1. Add model validation to persona routes

**`packages/server/src/routes/personas.ts`:**

```typescript
// In POST / (create) and PUT /:id (update):
const parsed = personaSchema.safeParse(body)
if (!parsed.success) {
  return c.json({ error: "Validation failed", details: parsed.error.issues }, 400)

// NEW: Validate model against Pi registry
```

### 2. Create `validatePersonaModel()` helper

**`packages/server/src/agent/piSession.ts`** (extend existing module):

```typescript
export async function validatePersonaModel(input: {
  provider?: string
  model?: string
}): Promise<{ valid: boolean; message?: string; suggestion?: string }> {
  const provider = input.provider || "anthropic"
  const modelId = input.model || "claude-sonnet-4-5"

  const registry = getModelRegistry()

  // Check if the model exists in Pi's registry
  const model = registry.find(provider, modelId)
  if (!model) {
    // Try to suggest a close match
    const available = await registry.getAvailable()
    const providerModels = available.filter(m => m.provider === provider)
    const suggestion = providerModels.length > 0
      ? `Available models for ${provider}: ${providerModels.map(m => m.id).join(", ")}`
      : `No models found for provider "${provider}". Check your API key configuration.`

    return { valid: false, message: `Model "${modelId}" not found for provider "${provider}".`, suggestion }
  }

  return { valid: true }
}
```

### 3. Make validation advisory, not blocking

For backward compatibility, model validation should warn but not block persona creation in all cases. A model might be temporarily unavailable (API key not configured yet). Strategy:

- **Create/Update:** Validate and return a warning in the response, but still save the persona
- **Task execution:** The runners already do auth pre-flight — they'll catch unavailable models at session start

```typescript
// In persona response, include validation status:
return c.json({
  ...persona,
  _meta: {
    modelValid: validation.valid,
    modelWarning: validation.valid ? undefined : validation.message,
  }
}, 201)
```
