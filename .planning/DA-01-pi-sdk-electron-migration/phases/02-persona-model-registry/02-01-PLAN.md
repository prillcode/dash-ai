# PLAN 02-01 — Rewrite Models Endpoint with Pi ModelRegistry

## Objective
Replace the hardcoded `DEFAULT_MODELS` + `models.json` file approach with Pi's `ModelRegistry`, so the models endpoint returns live, authenticated model availability from Pi's registry.

## Tasks

### 1. Rewrite `packages/server/src/routes/models.ts`

**Current:** Reads `~/.dash-ai/models.json` or returns a hardcoded `DEFAULT_MODELS` object. No auth checking — just lists model names.

**New:** Use Pi's `ModelRegistry` to return models with real availability status:

```typescript
import { Hono } from "hono"
import { getModelRegistry } from "../agent/piSession"

export const modelsRouter = new Hono()

modelsRouter.get("/", async (c) => {
  const registry = getModelRegistry()

  // Get all models (including those without API keys)
  const allModels = registry.getAllModels()  // or equivalent list method

  // Get models that have valid credentials
  const available = await registry.getAvailable()
  const availableIds = new Set(available.map(m => `${m.provider}/${m.id}`))

  // Group by provider
  const providers = groupByProvider(allModels, availableIds)

  return c.json({
    providers,
    authMethods: getAuthMethods(registry),
  })
})
```

**Response shape:**
```json

```

### 2. Remove `models.json` file dependency

- Delete the `DEFAULT_MODELS` constant
- Delete the `~/.dash-ai/models.json` file read
- The Pi `ModelRegistry` is the single source of truth for available models
- If Pi is not initialized (shouldn't happen after DA-01 Phase 01), return an empty list with a warning

### 3. Add auth method info to the response

The frontend's persona form needs to know which providers are configured so it can show warnings. Pi's `AuthStorage` knows this:

### 4. Keep the endpoint public (no auth required)

`GET /api/models` is currently before the `authMiddleware` in `index.ts`. Keep it that way — model listing doesn't require the Dash AI API token.

### 5. Frontend compatibility

The current frontend (`PersonaSelector`, `PersonaForm`) consumes the existing response shape. Update the response to maintain backward compatibility or update the frontend to match the new shape. Check:

- `packages/client/src/api/personas.ts` — how models are fetched and displayed
- `packages/client/src/components/personas/PersonaForm.tsx` — how provider/model dropdowns are populated

The new response has richer data (reasoning, context window, availability) — the frontend should use this to:
- Show only available models in dropdowns (or show all with unavailable ones grayed out)
- Show which providers need API key configuration
- Add reasoning/thinking level options per model capabilities
