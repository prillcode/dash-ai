# Phase 03 — Review Action UX Refresh

## Objective
Replace the current approve/reject-only post-coding interaction model with clearer actions that better reflect partial progress, blocked work, and follow-up coding requests.

## Context
- See `../BRIEF.md`
- See `../ROADMAP.md`
- Primary files likely include `packages/client/src/components/tasks/TaskActionBar.tsx`, `packages/client/src/pages/TaskDetailPage.tsx`, and any supporting task UI components.

## Tasks
1. Review the existing action model in task detail.
   - Identify where approve/reject is currently shown and what assumptions it encodes.
2. Design a clearer post-coding action set.
   - Determine which actions should exist after a coding run, such as:
     - approve/finalize
     - iterate coding / request follow-up work
     - reject/abandon
     - mark blocked (if needed)
3. Update task-detail messaging.
   - Make it clear that repo changes may already exist before final approval.
   - Improve guidance when the coder stops with a dependency gap or partial completion.
4. Keep the action model minimal.
   - Avoid turning the task page into a complex PM system.
   - Focus on the most common next actions users actually need.

## Verification
- After a partial coding run, users can clearly choose a follow-up action without abusing reject/retry.
- Completed tasks still have a simple approval/finalization path.
- Task detail messaging aligns with actual repo/task behavior.
- `pnpm build` passes.

## Done When
- The post-coding action model is more intuitive than approve/reject alone.
- Task detail messaging better matches iterative coding reality.
