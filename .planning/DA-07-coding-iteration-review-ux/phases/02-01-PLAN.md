# Phase 02 — Coding Runner Continuation Behavior

## Objective
Make codingRunner continuation/iteration runs aware of existing repo changes, the selected plan, and user feedback so follow-up coding sessions refine the work instead of restarting blindly.

## Context
- See `../BRIEF.md`
- See `../ROADMAP.md`
- Primary files likely include `packages/server/src/agent/codingRunner.ts` and `packages/server/src/services/queueWorker.ts`.

## Tasks
1. Audit the current coding prompt contract.
   - Identify what context is provided for fresh coding runs today.
   - Determine what additional feedback/continuation context is needed for iteration runs.
2. Design iteration-aware coding prompts.
   - Include user feedback, current selected execution doc, and instructions to continue from the current repo state.
   - Avoid prompting the coder to redo completed work unnecessarily.
3. Wire task feedback into queue/runner behavior.
   - Ensure the queue worker passes persisted coding feedback into codingRunner when appropriate.
   - Reset/clear feedback when the iteration run starts or completes as needed.
4. Improve iteration diagnostics.
   - Surface whether a coding run is fresh or iterative.
   - Include the follow-up feedback context in useful, concise form.

## Verification
- Iterated coding runs incorporate user feedback.
- The runner continues from the current repo state and selected plan doc coherently.
- Diagnostics clearly distinguish fresh coding from iteration.
- `pnpm build` passes.

## Done When
- Coding iteration runs behave intentionally and predictably.
- User feedback meaningfully shapes the next coding run.
