# Phase 01 — Coding Iteration Workflow and Status Model

## Objective
Define and implement the minimum task workflow changes needed so users can provide follow-up feedback after a coding run and continue coding coherently, while also splitting plan approval from execution so `READY_TO_CODE` does not auto-start coding.

## Context
- See `../BRIEF.md`
- See `../ROADMAP.md`
- Primary files likely include `packages/server/src/routes/tasks.ts`, `packages/server/src/services/taskService.ts`, `packages/server/src/db/schema.ts`, and `packages/client/src/components/tasks/TaskActionBar.tsx`.

## Tasks
1. Audit the current coding lifecycle states and actions.
   - Confirm how `PLANNED`, `READY_TO_CODE`, `QUEUED`, `RUNNING`, `AWAITING_REVIEW`, `APPROVED`, `REJECTED`, and `FAILED` are currently used.
   - Identify the smallest extension needed to separate approval from execution and to support iterative coding.
2. Design the pre-run vs execution split.
   - Keep `READY_TO_CODE` as the reviewed/approved pre-run state.
   - Introduce or repurpose an explicit queue/run trigger state/action so coding only starts after a deliberate user action.
3. Design coding iteration semantics.
   - Decide whether iteration should use an existing state (for example, returning to `READY_TO_CODE` / a new queue trigger) or introduce a narrow new state.
   - Decide what feedback needs to be persisted on the task.
4. Add server-side support.
   - Create a task action/endpoint for iterating coding with feedback.
   - Create or update the explicit action that queues/starts coding from `READY_TO_CODE`.
   - Persist the feedback and reset any fields needed for the next coding run.
5. Add initial UI support.
   - Add a coding iteration action in task detail.
   - Provide a feedback form similar in spirit to plan iteration.
   - Update the task actions so `READY_TO_CODE` does not immediately launch coding on its own.
6. Keep the UX and state transitions understandable.
   - Ensure labels match reality: code may already exist in the repo and the user may simply be asking for follow-up work.

## Verification
- A task can remain `READY_TO_CODE` without auto-starting until the user explicitly queues/runs it.
- A coding task that has partial progress can accept user feedback and re-enter coding coherently.
- Feedback is persisted and available to the next coding run.
- Existing approval/review flows still make sense for completed tasks.
- `pnpm build` passes.

## Done When
- `READY_TO_CODE` is a true pre-run approval state, not an automatic execution trigger.
- Coding iteration exists as a first-class action.
- The minimum status/persistence model is in place to support follow-up coding runs.
