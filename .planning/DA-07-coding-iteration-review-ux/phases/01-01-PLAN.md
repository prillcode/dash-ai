# Phase 01 — Coding Iteration Workflow and Status Model

## Objective
Define and implement the minimum task workflow changes needed so users can provide follow-up feedback after a coding run and continue coding coherently, without overhauling the entire task state machine.

## Context
- See `../BRIEF.md`
- See `../ROADMAP.md`
- Primary files likely include `packages/server/src/routes/tasks.ts`, `packages/server/src/services/taskService.ts`, `packages/server/src/db/schema.ts`, and `packages/client/src/components/tasks/TaskActionBar.tsx`.

## Tasks
1. Audit the current post-coding task states and actions.
   - Confirm how `AWAITING_REVIEW`, `APPROVED`, `REJECTED`, and `FAILED` are currently used.
   - Identify the smallest extension needed for iterative coding.
2. Design coding iteration semantics.
   - Decide whether iteration should use an existing state (for example, returning to `READY_TO_CODE` / `RUNNING`) or introduce a narrow new state.
   - Decide what feedback needs to be persisted on the task.
3. Add server-side support.
   - Create a task action/endpoint for iterating coding with feedback.
   - Persist the feedback and reset any fields needed for the next coding run.
4. Add initial UI support.
   - Add a coding iteration action in task detail.
   - Provide a feedback form similar in spirit to plan iteration.
5. Keep the UX and state transitions understandable.
   - Ensure labels match reality: code may already exist in the repo and the user may simply be asking for follow-up work.

## Verification
- A coding task that has partial progress can accept user feedback and re-enter coding coherently.
- Feedback is persisted and available to the next coding run.
- Existing approval/review flows still make sense for completed tasks.
- `pnpm build` passes.

## Done When
- Coding iteration exists as a first-class action.
- The minimum status/persistence model is in place to support follow-up coding runs.
