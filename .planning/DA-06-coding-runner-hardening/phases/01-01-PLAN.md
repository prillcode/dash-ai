# Phase 01 — Coding Entry Validation and UX Guardrails

## Objective
Prevent coding from starting unless an executable `PLAN.md` or `EXECUTION.md` actually exists, and remove or constrain UI/server paths that currently imply coding can safely begin without planning output.

## Context
- See `../BRIEF.md`
- See `../ROADMAP.md`
- Primary files likely include `packages/server/src/routes/tasks.ts`, `packages/server/src/services/taskService.ts`, `packages/client/src/components/tasks/TaskActionBar.tsx`, and `packages/client/src/components/tasks/PlanningSection.tsx`.

## Tasks
1. Audit current coding-entry paths.
   - Confirm all ways a task can become `READY_TO_CODE` or otherwise enter coding.
   - Identify which UI/server transitions currently bypass executable-plan validation.
2. Add server-side executable-plan validation.
   - Require a valid `planPath` plus at least one executable artifact (`PLAN.md` or `EXECUTION.md`) before allowing coding entry.
   - Return a clear 400-style error when coding is attempted without a valid plan.
3. Fix misleading UI actions.
   - Rework or disable “Skip Planning & Start Coding” when no executable plan exists.
   - Ensure buttons and messaging reflect actual backend rules.
4. Keep lifecycle behavior coherent.
   - Preserve legitimate coding starts for tasks that already have executable planning output.
   - Ensure task detail and planning sections communicate why coding is blocked when applicable.

## Verification
- Tasks without executable plans cannot be moved into coding flow.
- Tasks with valid `PLAN.md` / `EXECUTION.md` can still be queued normally.
- UI no longer suggests unsafe coding entry paths.
- `pnpm build` passes.

## Done When
- Coding entry is gated by a real executable-plan check.
- Unsafe coding-entry UI is removed or clearly constrained.
