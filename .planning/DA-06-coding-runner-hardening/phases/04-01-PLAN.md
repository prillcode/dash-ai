# Phase 04 — Planning-Artifact Guardrails and End-to-End Verification

## Objective
Add narrow `.planning` artifact guardrails for coding sessions and verify the full coding lifecycle end-to-end after the hardening work is complete.

## Context
- See `../BRIEF.md`
- See `../ROADMAP.md`
- This phase should happen after entry validation, cancellation, and no-op handling are in place.

## Tasks
1. Add narrow `.planning` guardrails.
   - Allow broad repo code changes where implementation requires them.
   - Restrict coding from reading/writing unrelated `.planning/*` work items by default.
   - Keep the selected work item’s planning artifacts available.
2. Verify real coding scenarios.
   - Valid plan → successful coding run
   - Valid plan → canceled coding run
   - Valid plan → no-op/empty-diff run
   - Invalid/no-plan task → blocked coding start
3. Audit user-facing diagnostics.
   - Ensure task detail/timeline surfaces reflect the improved coding behavior clearly.
   - Add minimal UX follow-through only if needed.
4. Finalize readiness.
   - Summarize what changed, what was verified, and any follow-up work.

## Verification
- Coding stays inside the intended work item’s planning artifacts.
- The main coding lifecycle paths behave coherently.
- `pnpm build` passes.

## Done When
- Coding guardrails are in place without blocking legitimate implementation work.
- End-to-end verification shows coding is safer and more inspectable.
