# Phase 02 — Real Cancellation and Session Lifecycle Control

## Objective
Make cancel/reset behavior actually control live coding sessions instead of only mutating task state in the database.

## Context
- See `../BRIEF.md`
- See `../ROADMAP.md`
- Primary files likely include `packages/server/src/services/queueWorker.ts`, `packages/server/src/agent/codingRunner.ts`, and relevant task action routes/UI.

## Tasks
1. Design active coding session tracking.
   - Add in-memory tracking keyed by task id for live coding sessions.
   - Ensure tracked sessions are cleaned up on success, failure, cancel, and timeout.
2. Wire cancel behavior to real aborts.
   - Differentiate queued-task cancellation from running-task cancellation.
   - Ensure canceling a running task aborts the actual Pi session.
3. Align task state handling.
   - Decide how canceled sessions should affect task status, error message, and timestamps.
   - Avoid stale sessions continuing to append events after cancellation.
4. Update UI/server affordances as needed.
   - Ensure the action labels and resulting behavior match what actually happens.

## Verification
- Canceling a running coding task aborts the live session.
- Canceling a queued task prevents it from starting.
- Session handles are cleaned up properly after termination.
- `pnpm build` passes.

## Done When
- Cancel means real cancel.
- Task/session state remains coherent across queued and running cancellation paths.
