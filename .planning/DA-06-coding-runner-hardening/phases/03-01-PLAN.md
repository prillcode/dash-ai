# Phase 03 — No-Op Run Handling and Coding Diagnostics

## Objective
Distinguish real implementation runs from no-op/empty-diff runs, and improve coding-session diagnostics so users and developers can tell what the runner actually did.

## Context
- See `../BRIEF.md`
- See `../ROADMAP.md`
- Primary files likely include `packages/server/src/agent/codingRunner.ts`, `packages/server/src/services/queueWorker.ts`, `packages/server/src/services/taskService.ts`, and task/timeline UI surfaces.

## Tasks
1. Define no-op behavior.
   - Decide how empty-diff runs should surface: failure, explicit no-op outcome, or guarded review state.
   - Keep the behavior understandable to users.
2. Persist coding metadata earlier.
   - Store session id and selected execution doc early enough for live debugging.
   - Ensure failure paths still preserve useful metadata.
3. Improve coding summary/diagnostics.
   - Add clear summary events/messages for selected doc, diff presence, and session outcome.
   - Expose enough information in the UI to inspect coding behavior without digging through raw logs.
4. Keep success criteria honest.
   - Prevent truly no-op runs from looking identical to successful implementation runs.

## Verification
- Empty-diff sessions are distinguishable from successful implementation runs.
- Session id and selected execution doc are visible before task completion.
- Coding summaries remain understandable in the timeline/UI.
- `pnpm build` passes.

## Done When
- No-op coding runs are surfaced explicitly.
- Early coding metadata is available for debugging.
- Successful implementation runs are easier to distinguish from no-op outcomes.
