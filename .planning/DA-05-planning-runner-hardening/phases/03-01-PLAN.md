# Phase 03 — Session Telemetry and Event Noise Reduction

## Objective
Improve runner observability so long planning/coding sessions are easier to interpret, while reducing low-value event/text noise and preserving enough detail to debug failures and targeting decisions.

## Context
- See `../BRIEF.md`
- See `../ROADMAP.md`
- Likely implementation files: `packages/server/src/services/eventService.ts`, `packages/server/src/agent/planningRunner.ts`, `packages/server/src/agent/codingRunner.ts`, and possibly relevant client task-detail surfaces.
- Current sessions generate many `PLANNING_TEXT` events, which makes diagnosis possible but noisy.

## Tasks
1. Identify the minimum useful diagnostics.
   - Define which fields matter most during planning/coding: task id, mode, expected target prefix, detected target directory, selected execution doc, runtime, turn counts, tool counts, and failure reasons.
2. Design event-noise reduction strategy.
   - Decide whether to coalesce streamed text, suppress some intermediate narration, or chunk text updates before persistence/broadcast.
   - Avoid losing critical breadcrumbs such as tool usage and final status.
3. Implement telemetry improvements.
   - Add or improve events around planning target selection and coding execution doc selection.
   - Add summary-style events where they reduce the need to inspect hundreds of deltas.
4. Validate client/debugging usability.
   - Confirm the resulting event stream remains readable in the current UI/API usage.
   - If necessary, add a minimal UI affordance to surface key session metadata without dumping raw deltas.
5. Verify that debugging still works.
   - Ensure failures still leave enough evidence to understand what happened.
   - Confirm event persistence/broadcast behavior remains stable.

## Verification
- Similar planning/coding runs produce fewer or more useful events than before.
- Key target-selection details are visible without inspecting raw prompt text.
- Failures remain diagnosable from persisted events.
- Any minimal UI follow-through still builds and behaves correctly.
- `pnpm build` passes.

## Done When
- Event streams are less noisy or more digestible.
- Key runner decisions are surfaced explicitly.
- Debuggability is preserved or improved.
