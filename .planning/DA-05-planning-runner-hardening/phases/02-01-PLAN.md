# Phase 02 — Planning Speed and Mode Right-Sizing

## Objective
Reduce planning runtime for straightforward tasks by introducing lighter-weight planning behavior, better planning-mode selection, and more intentional model/thinking defaults while preserving a detailed path for ambiguous or multi-phase work.

## Context
- See `../BRIEF.md`
- See `../ROADMAP.md`
- Primary implementation files likely include `packages/server/src/agent/planningRunner.ts`, plus any route/settings surfaces needed to choose or infer planning mode.
- Current planning always performs scaffold + phase deepening, which is reliable but expensive.

## Tasks
1. Define planning modes and selection rules.
   - Decide whether the system should support modes such as fast/lightweight, detailed/full, and optional research-heavy planning.
   - Decide whether mode is inferred from task complexity, persona, or an explicit user choice.
2. Design the lightweight path.
   - Determine when fresh planning can stop after scaffold/roadmap generation.
   - Decide whether lightweight mode should create `EXECUTION.md`, only `BRIEF.md` + `ROADMAP.md`, or a reduced phase-plan set.
3. Review model and thinking defaults.
   - Evaluate whether planning should use a lighter thinking level or a faster default model for common tasks.
   - Preserve the ability for planning personas to opt into deeper planning when needed.
4. Implement the chosen mode behavior.
   - Update runner prompt flow and/or task creation/planning APIs as needed.
   - Ensure the produced plan artifacts remain compatible with the rest of the task lifecycle.
5. Capture timing comparisons and quality checks.
   - Compare runtime and output usefulness for at least one straightforward task and one more ambiguous task.
   - Ensure the faster path still generates acceptable, actionable plan artifacts.

## Verification
- Straightforward planning tasks complete faster than the current baseline.
- Detailed planning remains available for complex work.
- Produced artifacts are still readable and compatible with task review/coding flows.
- Any new mode selection logic behaves predictably.
- `pnpm --filter server build` passes.

## Done When
- The system has a clear, intentional planning-mode strategy.
- Straightforward tasks can avoid unnecessary deepening work.
- Planning speed improves without unacceptable quality loss.
- Mode behavior is understandable to developers and users.
