# Phase 01 — Planning Targeting Guardrails

## Objective
Make fresh planning deterministic about the intended work item by tightening prompt scope, validating that the expected planning directory was created, and preventing normal planning runs from reading unrelated `.planning/*` work items unless explicitly allowed.

## Context
- See `../BRIEF.md`
- See `../ROADMAP.md`
- Primary implementation file: `packages/server/src/agent/planningRunner.ts`
- Recent fixes already switched planning to write-capable tools and improved target detection, so this phase should build on that baseline rather than redo it.

## Tasks
1. Audit the current planning prompt contract and target-detection logic.
   - Confirm exactly how fresh planning, iteration, and fallback detection currently behave.
   - Identify where prompt text still leaves room for precedent fishing or unrelated `.planning` reads.
2. Strengthen fresh-planning prompt boundaries.
   - Explicitly state the intended identifier, expected work item prefix, and target `.planning` location.
   - Tell the planner not to read other `.planning/*` work items during normal planning.
   - Distinguish allowed repo/codebase exploration from forbidden historical plan exploration.
3. Add hard matching-directory validation for fresh planning.
   - Require a created directory matching the expected identifier/work-name prefix.
   - If no matching directory is created, return a clear planning failure instead of falling back to the latest directory.
4. Tighten iteration behavior.
   - Ensure iteration only reads and updates the existing target work item.
   - Preserve the current ability to refine the same work item in place.
5. Add concise diagnostics for planning target decisions.
   - Emit an event or log payload showing expected target prefix, detected target directory, and whether planning was fresh vs iteration.
6. Verify error messaging and state handling.
   - Ensure user-facing/task-facing failure behavior is clear if scaffold creation fails.
   - Confirm task status behavior remains coherent when planning fails fast.

## Verification
- A new task with a unique identifier creates a matching `.planning/<identifier>-...` directory.
- Normal planning traces show code/context reads but no unrelated `.planning/*` reads.
- A forced or simulated scaffold failure returns an explicit error instead of reusing another work item.
- Iteration updates only the existing target work item.
- `pnpm --filter server build` passes.

## Done When
- Fresh planning requires a matching target directory.
- Planning no longer silently reuses another work item on scaffold failure.
- Normal planning is explicitly constrained away from unrelated `.planning/*` artifacts.
- Planning diagnostics make target selection easy to inspect.
