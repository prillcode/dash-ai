# PLAN 02-04 — Watch Command (Live Events)

## Objective
Stream real-time agent events to the terminal during planning and coding sessions. This is the human-facing UX payoff — watching the AI work live from the terminal.

## Tasks

### 1. `dash-ai tasks watch <id>`

Connects to the task's event stream and renders activity in real-time:

```typescript
tasks.command("watch <id>")
  .description("Stream live events for a running task")
  .option("--follow", "keep watching after task completes (default: exit on completion)")
  .action(async (id, opts) => {
    const ctx = getContext(opts)
    const task = await client.get(`/api/tasks/${id}`)

    if (!["IN_PLANNING", "RUNNING", "QUEUED", "READY_TO_CODE"].includes(task.status)) {
      error(`Task is not running (status: ${task.status}). Nothing to watch.`, { exitCode: 1 })
    }

    // Connect to WebSocket event stream
    const ws = new WebSocket(`ws://localhost:${port}/ws/tasks/${id}/stream`)

    ws.on("message", (data) => {
      const event = JSON.parse(data.toString())
      renderEvent(event, ctx)
    })

    ws.on("close", () => {
      if (!opts.follow) {
        info("Session ended.")
        process.exit(0)
      }
    })

    // Graceful exit
    process.on("SIGINT", () => {
      ws.close()
      info("\nStopped watching. Task continues running server-side.")
      process.exit(0)
    })
  })
```

### 2. Event rendering (`src/output/events.ts`)

Format agent events for terminal display:

```typescript
export function renderEvent(event: AgentEvent, ctx: CliContext): void {
  if (ctx.json) {
    // NDJSON output for agent consumers
    process.stdout.write(JSON.stringify(event) + "\n")
    return
  }

  switch (event.type) {
    case "PLANNING_EVENT":
    case "CODING_EVENT":
      if (event.status === "starting") {
        console.log(chalk.blue(`⏳ ${event.message}`))
      } else if (event.status === "completed") {
        console.log(chalk.green(`✓ ${event.message}`))
      }
      break

    case "PLANNING_TEXT":
    case "CODING_TEXT":
      process.stdout.write(chalk.gray(event.delta))
      break

    case "TOOL_START":
      console.log(chalk.cyan(`  → ${event.toolName}`) + chalk.gray(` ${truncate(JSON.stringify(event.args), 80)}`))
      break

    case "TOOL_END":
      if (event.isError) {
        console.log(chalk.red(`  ✗ ${event.toolName} (error)`))
      } else {
        console.log(chalk.green(`  ✓ ${event.toolName}`))
      }
      break

    case "TURN_START":
      console.log(chalk.bold(`--- Turn ${event.turnIndex} ---`))
      break

    case "TURN_END":
      console.log()  // blank line between turns
      break

    case "ERROR":
      console.log(chalk.red(`⚠ ${event.message}`))
      break
  }
}
```

### 4. `--json` mode for watch (NDJSON)

For agent consumers, `watch --json` emits newline-delimited JSON events:

```
{"type":"PLANNING_EVENT","status":"starting","message":"Starting planning session"}
{"type":"TOOL_START","toolName":"read","args":{"path":"src/api/auth.ts"}}
{"type":"TOOL_END","toolName":"read","isError":false}
{"type":"PLANNING_TEXT","delta":"The task requires adding rate limiting..."}
{"type":"PLANNING_EVENT","status":"completed","message":"Planning complete"}
```

An agent consumer can pipe this into its own processing pipeline.

### 5. Connection via thin client or embedded

The watch command should work in both modes:
- **Thin client:** connects WebSocket to `DASH_AI_URL` (the remote server where the task is running)
- **Embedded:** if the task is being planned/coded by the embedded server, connect to its WebSocket

In practice, `watch` will usually be in thin client mode — you start a plan in one terminal, then `watch` from another (or from a script). But it should also work when the embedded server is doing the work.

### 6. Task completion detection

When the task reaches a terminal state (PLANNED, AWAITING_REVIEW, COMPLETE, FAILED), the watch command:
- Prints the final status
- Shows a summary (duration, tool calls count, etc.)
- Exits with appropriate code (0 for success states, 6 for FAILED)
