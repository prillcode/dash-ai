# PLAN 02-02 — Task CRUD Commands

## Objective
Implement all read-only and data-manipulation CLI commands. After this plan, a user can manage projects, personas, and tasks from the terminal, with both human-readable table output and `--json` mode.

## Tasks

### 2. Task list command (`src/commands/tasks.ts` — partial)

```typescript
tasks.command("list")
  .description("List tasks")
  .option("--status <status>", "filter by status (DRAFT, PLANNED, RUNNING, etc.)")
  .option("--project <name>", "filter by project name")
  .option("--persona <id>", "filter by persona")
  .action(async (opts) => {
    const { client } = await resolveClient(getContext(opts))
    const params = new URLSearchParams()
    if (opts.status) params.set("status", opts.status)
    if (opts.project) params.set("projectName", opts.project)

    const tasks = await client.get(`/api/tasks?${params}`)

    formatOutput(ctx, tasks, () => {
      // Table: ID | Title | Status | Priority | Project | Updated
      return formatTable(
        ["ID", "Title", "Status", "Priority", "Project", "Updated"],
        tasks.map(t => [t.id.slice(0, 8), truncate(t.title, 40), t.status, t.priority, t.projectName ?? "—", relativeTime(t.updatedAt)])
      )
    })
  })
```

### 3. Task create command

```bash
dash-ai tasks create \
  --project my-app \
  --title "Add rate limiting to API" \
  --description "Add rate limiting middleware with 100 req/min per IP" \
  --planner "Planning Expert" \
  --coder "Coding Agent" \
  --json
```

Implementation:
- Resolve project name to `projectId` (look up via `GET /api/projects?name=` or list + filter)
- Resolve persona names to IDs (look up via `GET /api/personas`)
- `POST /api/tasks` with full payload
- Print created task details (table or JSON)

### 4. Task show command

```bash
dash-ai tasks show abc-123 [--json]
```

Display full task details including:
- Title, description, status, priority
- Project name + path
- Planning persona, coding persona
- Plan path (if planned)
- Session ID (if running/complete)
- Diff path (if awaiting review)
- Key timestamps (created, started, completed)

### 7. Config command

```bash
dash-ai config list                              # show: URL, token (masked), DB path, pi agent dir
dash-ai config set DASH_AI_URL http://localhost:3000
dash-ai config set DASH_AI_TOKEN xxx
```

Config is stored in `~/.dash-ai/config.json`. This is separate from the server's `models.json` and database — it's CLI-specific connection config.

Actually, for V1, config can just read from env vars. Skip the file-based config — `DASH_AI_URL` and `DASH_AI_TOKEN` env vars are sufficient. Add a `config list` that shows what's resolved.

### 8. Error handling

All commands should:
- Catch API errors and display human-readable messages (not raw HTTP errors)
- Set exit codes properly (0 success, 3 not found, 4 auth, 5 unreachable)
- In `--json` mode, output `{ "error": "message", "code": 3 }` and exit with the code

```typescript
try {
  const result = await client.get(`/api/tasks/${id}`)
  // ... format output
} catch (err) {
  if (err.status === 404) {
    error(`Task not found: ${id}`, { exitCode: 3 })
  } else if (err.status === 401) {
    error(`Authentication failed. Set DASH_AI_TOKEN or use --token.`, { exitCode: 4 })
  } else {
    error(`Request failed: ${err.message}`, { exitCode: 1 })
  }
}
```

### 9. Auto-cleanup for embedded mode

When running in embedded mode, the CLI should tear down the server when the command completes:

```typescript
const { client, embedded } = await resolveClient(ctx)
try {
  // ... run command
} finally {
  if (embedded) {
    await teardownEmbeddedServer()
  }
}
```

Wrap this in a shared `withClient()` helper to avoid repeating in every command handler.
